# Too Many Tinkers 项目说明

## 项目目标

本项目是面向 Minecraft 1.12.2 / Cleanroom / Java 25 / LWJGL3 / opengl4.6 的匠魂 2 渲染优化项目。

核心目标不是保持匠魂原版渲染兼容，而是尽量接管 TiC2 工具和部件的模型、材质、贴图生成与 item 渲染路径，降低以下开销：

- 匠魂原版按“部件 x 材质”生成大量动态贴图导致的内存波动。
- 原版 baked quad / item model override 带来的对象分配和缓存压力。
- 大量工具或材料组合下的 CPU draw call 和模型构建成本。

当前方向是：**用少量 TMT 自有动态大图 + GPU shader 采样 + VBO/SSBO/MDI 渲染替代 TiC2 原版动态材质贴图生成。**

## 接管范围

当前项目已经接管 TiC2 工具和部件的主要客户端渲染链路：

- `CustomTextureCreator.createMaterialTextures`：在 HEAD 取消原版材质贴图生成，改为构建 TMT material/mask 描述。
- `MaterialModel.bakeIt`：返回 TMT 的 part model。
- `ToolModel.bake`：返回 TMT 的 GPU tool template model。
- `RenderItem.renderItem/renderModel`：识别 `TmtGpuItemModel` 后直接走 TMT GPU renderer，并取消原渲染。

项目仍会读取 TiC2 的注册数据和资源元数据，例如：

- `TinkerRegistry.getAllMaterials()`
- `MaterialRenderInfo`
- TiC 的 base texture 列表
- 原始 PNG 资源

这些读取只作为输入数据，不再触发 TiC2 原版的批量动态贴图生成。

## 当前渲染思路

渲染数据被拆成三类：

1. **Part mask map**
   - 管理类：`TmtPartMaskMapManager`
   - 内容：部件基础灰度/透明度 mask、suffix mask、modifier/override 动态贴图 mask。
   - 用途：shader 根据 `maskSlot + local UV` 采样形状、灰度和 alpha。
   - 注意：modifier 和 override 贴图可能不在 TiC `baseTextures` 里，因此当前支持按需注册 mask。

2. **Material map**
   - 管理类：`TmtMaterialMapManager`
   - 内容：材质颜色、ramp、源纹理采样块。
   - 当前类型：
     - `TYPE_DIRECT`
     - `TYPE_SOLID`
     - `TYPE_RAMP`
     - `TYPE_TEXTURE`
   - 用途：shader 根据 material type/index 把 mask 灰度映射成最终颜色。

3. **Instance / indirect command**
   - instance 数据写入 `TmtInstanceBuffer` 的 SSBO。
   - draw command 写入 `TmtIndirectCommandBuffer`。
   - 最终由 `glMultiDrawElementsIndirect` 提交。

## Mesh 管线

`PartSpriteMeshCache` 现在不再给每个 part mesh 创建独立 VAO/VBO/EBO。

当前做法：

- 所有 part mesh 合并到一个全局 mega VAO/VBO/IBO。
- 每个 `PartMesh` 只保存：
  - `id`
  - `firstIndex`
  - `indexCount`
  - `baseVertex`
- renderer 按 mesh id 聚合 instance 数量，生成 MDI command。

这减少了 per-mesh GL 对象数量和每帧 draw call 循环。旧的 `glDrawElementsInstanced` 路径已经被替换为 `glMultiDrawElementsIndirect`。

## Layer 与材质解析

`TmtToolRenderDescriptor.Layer` 创建时会提前解析：

- `maskTexture`
- `maskSlot`
- `materialType`
- `materialIndex`
- `flags`

这样渲染阶段不再反复用 material id 字符串查 `MaterialDescriptorRegistry`。

工具本体和 broken 替换层保持同级；modifier 层使用很小的正 `zBias`，保证渲染在工具上层。

## Shader 要求

当前 shader 使用：

- OpenGL 4.3
- `GL_ARB_shader_draw_parameters`
- SSBO
- `gl_BaseInstanceARB`

`gl_BaseInstanceARB` 用于从 MDI command 的 `baseInstance` 定位 SSBO 中对应 instance 数据。

如果目标机器缺少 `GL_ARB_shader_draw_parameters`，初始化 shader 时会直接报错。

## 缓存策略

当前缓存重点：

- `TmtGpuToolTemplateModel`：用轻量 key 缓存工具 descriptor，避免完整 NBT copy。
- `TmtPartMaterialModel`：按 material id 缓存 part stack model。
- `PartSpriteMeshCache`：按 mask texture、offset、rotation、zBias 缓存 mesh range。
- `MaterialDescriptorRegistry.rebuild` 后会主动清理相关缓存。

注意：工作树中可能还有用户已有改动，不要随手 reset 或 checkout。

## 已知注意点

- modifier / override 贴图必须能进入 mask map，否则会表现为紫色 missing 方块。
- mask map 是动态纹理，按需注册新 mask 后，必须在绑定纹理前完成注册，否则这一帧 shader 可能拿到旧高度或旧纹理。
- MDI 会按 mesh command 聚合绘制。涉及透明层或强顺序需求时，需要通过 zBias 或拆 command 保证表现。
- `TmtPartDefinition.singleTexture(texture, zBias)` 的 zBias 已经生效，不要再把它当作无用参数。
- fastutil 不需要在项目里额外声明依赖，当前环境已经提供。

## 后续可继续优化

优先级较高的方向：

- 把 instance buffer 和 indirect command buffer 改成 persistent mapped ring buffer，减少 `glBufferSubData` 同步和拷贝。
- 继续把 hot path 的字符串 material id 压成 int id。
- 对 material map / mask map 做内容 hash 去重，避免重复 mask 或重复 ramp。
- 给 material map / mask map 加 debug dump 开关，方便定位紫块、错色、复用错误。
- 清理 MDI 初版引入的结构膨胀，把临时 helper 收敛成更小的内部实现。

## 验证方式

基础编译：

```powershell
.\gradlew.bat compileJava
```

游戏内重点看：

- 工具正常材质颜色是否正确。
- broken 工具是否和正常工具层级一致。
- modifier 是否稳定压在工具上层。
- modifier / override / 动态变化贴图是否还有紫块。
- 大量材料和工具切换时内存波动是否明显下降。
